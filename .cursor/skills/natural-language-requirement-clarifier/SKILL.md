# 自然语言需求澄清器

## 目标

在执行开发前，先把用户需求从“模糊描述”转成“可执行任务”。

## 适用场景

- 用户只说想法，没有明确范围或验收标准。
- 用户描述含糊，可能有多种实现路径。
- 用户同时提到多个目标，优先级不明确。

## 标准流程

1. **提炼原始需求**
   - 用 1-3 句话复述用户想要的结果。
   - 标记不确定点（范围、优先级、约束、验收标准）。

2. **拆解执行项**
   - 拆成可交付小任务（页面/逻辑/数据/测试/文档）。
   - 指出依赖关系和先后顺序。

3. **补全约束**
   - 技术约束（语言、框架、版本、兼容性）。
   - 业务约束（文案、交互、性能、上线风险）。
   - 边界（明确“不做什么”）。

4. **必要时请求确认**
   - 优先用结构化选项（单选/多选）让用户确认关键分歧。
   - 如果工具支持，使用问答表单形式；否则用简洁选项列表。

5. **确认后再执行**
   - 给出“执行版任务单”：目标、步骤、验收、风险。
   - 获得确认后进入实现阶段。

## 输出模板

```markdown
需求理解：
- [一句话目标]

待确认项：
- [分歧点A：选项1/选项2]
- [分歧点B：选项1/选项2]

执行版任务单：
- 目标：
- 约束：
- 步骤：
- 验收标准：
```

## 质量要求

- 不擅自扩需求。
- 不在关键歧义未确认前直接大改代码。
- 每次确认只问最关键的 1-3 个问题，减少沟通负担。
