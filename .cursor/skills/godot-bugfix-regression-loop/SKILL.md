# 问题修复与回归闭环

## 目标

把修 bug 变成标准流程，避免“修了A坏了B”。

## 固定流程

1. 复现问题：记录触发步骤、预期行为、实际行为。
2. 定位根因：找到最小故障路径（脚本/信号/节点/输入）。
3. 最小修复：只改必要代码，不做大范围重构。
4. 回归验证：按清单覆盖关联路径。

## 防卡死执行规则（新增）

- 若单次工具调用或单轮推理 **超过 90 秒** 无新进展，必须立刻中断当前路径并回报状态。
- 中断后执行“二选一”：
  1) 用更小粒度重试（拆成更小补丁/更短命令）；  
  2) 切换到备用方案（例如只读排查、先做最小修复）。
- 连续两次重试仍无进展时，不得继续阻塞，必须向用户同步“当前卡点 + 下一步可执行方案”。
- 禁止长时间停留在同一条命令/补丁上不反馈。

## 子代理监控模板（可选）

- 当任务跨度大或容易卡住时，启动一个后台子代理做“节奏监控”：
  - 子代理类型：`godot-tech-troubleshooter` 或 `generalPurpose`
  - 目标：每轮输出“已完成/卡点/下一步”，督促主流程及时切换策略
- 监控子代理不负责大改代码，只负责识别阻塞并建议重试路径。

## Godot 专项检查

- 场景切换后节点是否仍有效（`is_instance_valid`）。
- 信号是否重复连接或未断开。
- 输入冲突（地图/对话/攻击/移动端按钮）。
- 云端与单机分支是否都验证过。

## 输出格式

- 问题描述（含复现）
- 根因说明
- 修复点与风险
- 回归清单与结果

## 回归最小清单

- [ ] 大厅进入世界
- [ ] 世界返回大厅
- [ ] 与 NPC 交互
- [ ] 地图开关
- [ ] 攻击与技能
- [ ] 试炼进入与返回（若相关）
