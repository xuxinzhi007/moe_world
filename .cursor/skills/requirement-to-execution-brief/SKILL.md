# 需求转执行任务单

## 目标

在编码前产出统一任务单，确保开发目标与验收标准一致。
并强制任务满足工程化交付（非碎片化）。

## 任务单结构

1. 目标（做成什么）
2. 范围（做什么 / 不做什么）
3. 约束（技术、交互、文案、性能）
4. 步骤（按先后顺序）
5. 验收（可验证标准）
6. 风险（可能回归点）
7. 架构约束（复用现有模块，不新增平行系统）
8. 闭环要求（入口->行为->结果->回退）

## 输出模板

```markdown
执行任务单

目标：
- ...

范围：
- 做：...
- 不做：...

约束：
- ...

实施步骤：
1. ...
2. ...

验收标准：
- ...

风险与回归点：
- ...

架构约束：
- 复用现有场景/autoload，不创建重复系统

闭环要求：
- [进入] -> [执行] -> [结算/结果] -> [返回/恢复]
```

## Godot 项目补充

- 若涉及世界/试炼，默认双场景验证。
- 若涉及 UI，默认 PC + 移动端都核对。
- 若无法形成闭环，不进入开发阶段，先补需求。
