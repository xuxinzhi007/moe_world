# AI 游戏工程协议（Godot 4.4）

> 适用范围：本项目所有 AI 开发任务（代码、场景、UI、文档、测试）。

---

## 1. 目标

避免“东拼西凑”式开发，强制 AI 以**完整工程思维**交付：  
- 结构清晰  
- 主流程闭环  
- 可回归、可维护  
- 可持续扩展

---

## 2. 核心原则（必须遵守）

1. 先架构后功能：先确认系统边界与目录归属，再写代码。  
2. 先闭环后扩展：先打通进入/战斗/结算/返回，再加内容。  
3. 最小改动优先：不为“看起来更好”做大重构。  
4. 同核异皮：世界与试炼共享核心战斗逻辑，仅规则/场景差异化。  
5. 文档与代码同步：每次功能变更必须同步回归清单与执行看板。

---

## 3. 禁止事项（反碎片化）

- 禁止只给“孤立函数”而不说明模块边界。  
- 禁止新增与现有架构平行的重复系统。  
- 禁止绕过现有 Autoload/场景契约直接硬塞逻辑。  
- 禁止只改代码不更新测试点。  
- 禁止“理论建议”替代可执行任务单。
- 禁止臆造 Godot API、节点属性、枚举常量或信号名。
- 禁止臆造后端接口路径、字段结构、返回格式。

---

## 4. 标准交付格式（每个任务都要有）

1. 目标：本次要完成什么。  
2. 范围：改哪些文件，不改哪些。  
3. 实施：按步骤说明改动。  
4. 验收：可操作的手测项。  
5. 风险：潜在回归点与兜底方案。
6. API 校验：本次使用的 Godot API / 项目接口 / 后端接口的来源说明。

---

## 4.1 API 真实性约束（新增）

1. Godot 引擎 API 必须以 **Godot 4.4 官方文档** 为准。  
2. 项目内接口（Autoload、公共方法、信号）必须先在代码库中检索确认后再调用。  
3. 网络接口必须基于现有实现或后端文档，不可凭空补字段。  
4. 若接口真实性不确定，先标记为“待确认”，不得直接写入主流程代码。

---

## 5. 开发顺序（固定）

1) 需求澄清（是否闭环，是否可验收）  
2) 任务单（执行范围+验收标准）  
3) 实施改动（最小必要）  
4) 质量门禁（契约/路径/输入/分支）  
5) 更新文档（执行看板+测试清单）

---

## 6. 本项目工程锚点（Source of Truth）

- `docs/ARCHITECTURE.md`：架构与目录。  
- `docs/FLOW_UI_DESIGN_BLUEPRINT.md`：UI 页面与流程蓝图。  
- `docs/PRODUCT_MANAGER_GAME_WORLD_PLAN.md`：版本目标与玩法路线。  
- `docs/SUBAGENT_EXECUTION_BOARD.md`：子代理执行状态。  
- `docs/OPTIMIZATION_TEST_CHECKLIST.md`：回归测试清单。

---

## 7. 任务验收最低标准（DoD）

- Godot 4.4 GDScript 语法正确。  
- 主流程可走通（大厅 -> 世界/试炼 -> 返回）。  
- 相关场景（世界/试炼、PC/移动）都做回归说明。  
- 无新增高风险回归项。  
- 文档已同步。
