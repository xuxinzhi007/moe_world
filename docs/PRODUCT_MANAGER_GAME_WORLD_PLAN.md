# Moe World 游戏产品经理规划文档

> 目标：为当前项目建立一个可长期复用的“产品经理工作框架”，用于规划游戏世界的功能、玩法与版本节奏，并与研发/美术/音频协同推进。

---

## 1. 产品经理角色定义（本项目）

### 1.1 角色名称

- 游戏产品经理（Game PM）

### 1.2 核心职责

- 定义产品愿景与阶段目标（先可玩，再可持续，再可扩展）。
- 将“想法”拆解为可执行需求：功能、玩法、UI、数值、体验目标。
- 维护优先级与版本范围，避免范围蔓延。
- 建立验收标准（Definition of Done），确保每次迭代可测试、可回归。
- 跟踪玩家体验指标：留存、战斗节奏、成长反馈、操作流畅度。

### 1.3 本项目 PM 工作原则

- 小步快跑：每个版本都要“可进入、可战斗、可结算、可回退”。
- 同核异皮：世界模式与试炼模式共享核心战斗逻辑，只在场景和规则上差异化。
- 先稳定再增强：优先修复启动错误、主流程断点、性能热点，再做玩法扩展。
- 数据驱动：所有玩法调整必须绑定观察指标和测试结论。
- 工程化交付：严格遵循 `docs/AI_GAME_ENGINEERING_PROTOCOL.md`，禁止碎片化需求与碎片化代码。

## 2. 产品愿景与目标用户

### 2.1 一句话愿景

- 打造一个“轻量、爽快、可持续成长”的 2D 萌系世界探索与战斗游戏。

### 2.2 目标用户画像

- 喜欢短时高反馈战斗（3-10 分钟一局）的玩家。
- 喜欢角色成长与职业切换（战士/弓手/法师/牧师）的玩家。
- 喜欢轻社交（聊天、同场可见）但不强社交负担的玩家。

### 2.3 成功标准（阶段）

- P1（可玩）：新玩家 5 分钟内能完成“进入世界 -> 战斗 -> 获得成长反馈”。
- P2（可留）：玩家可理解成长路径，连续游玩 20 分钟不迷失。
- P3（可扩）：玩法可按模块扩展（副本、掉落、任务、活动）且回归成本低。

## 3. 游戏世界功能地图（PM 视角）

## 3.1 主流程模块

- 大厅：入口、身份、模式选择。
- 大世界：探索、战斗、掉落、交互。
- 试炼副本：怪潮压测、成长收益、结算反馈。
- 成长系统：职业、属性、技能、经验。
- UI/HUD：战斗信息、操作反馈、地图、背包、商店。
- 社交轻交互：聊天、在线玩家可见（云端场景）。

### 3.2 功能分层（必须有 / 应该有 / 可以有）

- 必须有（MVP）
  - 可稳定启动与切换场景。
  - 基础战斗闭环（攻击、受伤、击杀、经验、升级）。
  - 至少一个稳定副本（试炼）可重复游玩并结算。
  - 基础成长面板与关键 HUD 信息可见。
- 应该有（V1）
  - 职业差异化强化（技能手感、反馈、数值定位）。
  - 掉落体系与背包/商店联动。
  - 地图与探索信息（方位、目标、传送点引导）。
- 可以有（V2+）
  - 任务系统、周常挑战、活动事件。
  - 更完整联机协作目标（组队/共斗目标）。
  - 赛季化成长或主题副本轮换。

### 4.2 示例：试炼模式（简版）

## 5. 版本路线图（建议）

## P0 稳定性（当前阶段，立即）

- 修复启动报错和场景切换断点。
- 修复输入冲突（移动端与键盘切换）。
- 修复信号生命周期问题（进出场景重复连接）。
- 处理高频性能热点（判定、HUD、重绘、分组查询）。

## P1 核心体验打磨（1-2 周）

- 职业战斗手感统一与差异强化（命中反馈、技能节奏）。
- 试炼和世界“同核异皮”参数体系整理。
- 成长路径清晰化（升级、属性点、技能收益可感知）。

## P2 内容扩展（2-4 周）

- 掉落生态（材料、装备胚子、消耗品）。
- 背包-商店-成长联动闭环。
- 地图探索目标（热点区域、精英怪/事件点）。

## P3 长线系统（后续）

- 任务线、周常、挑战榜或赛季目标。
- 联机目标深化（协作挑战而非仅同屏）。

## 6. 验收标准（Definition of Done）

每个需求完成必须满足：

- 功能可运行：主路径无阻断。
- 语法与脚本安全：Godot 4.4 GDScript 无明显语法问题。
- 冒烟可通过：大厅 -> 世界 -> 试炼 -> 返回 全流程可走通。
- 性能可接受：高怪量场景操作无明显“卡住/失控”。
- 文档已同步：测试清单新增对应回归项。

## 7. 协同分工（你与 AI 团队）

### 你负责（建议）

- 美术资源、音频资源、最终体验主观评估。

- 风险前置识别（性能热点、状态机问题、信号生命周期）。
- 回归清单维护和文档更新。
- 根据你的反馈快速迭代修复。

## 8. 每周执行节奏（轻量流程）

- 周一：确定本周 3-5 个最高优先级目标（必须可验收）。
- 周中：每日同步“已完成/阻塞/风险”。
- 周末：统一冒烟 + 问题分级（严重/高/中/低） + 下周排期。

## 10. 附：产品经理快速判断清单

- 这个功能是否让玩家更快进入“有反馈”的状态？
- 这个改动是否会提高主流程稳定性？
- 这个系统是否会增加未来维护复杂度？
- 如果本周只能做一半，哪一半最有价值？
- 这个需求是否写清楚了可测试的验收条件？
